# migrate-start — 레거시 분석 + 스펙 초안 (구현 없음)

인자: `<기능명> <레거시-repo-경로>` — 비어 있으면 먼저 사용자에게 물어본다.

레거시 기능 이관의 분석·스펙 단계를 수행한다. **이 커맨드에는 구현 단계가 없다.**
구현은 사용자가 스펙을 승인한 뒤 `/migrate-implement`를 실행할 때만 진행된다.

[절대 규칙]
1. 레거시 경로는 read-only다. 어떤 파일도 수정하지 않는다.
2. 코드/쿼리에서 확인한 사실만 기록한다. 확인 못 한 동작은 "미확인"으로 남기고 절대 추측하지 않는다.
3. 이 단계에서는 production 코드를 한 줄도 작성하지 않는다.
4. 결제/인증/개인정보/공유 코드가 관련되면 멈추고 보고한다 (`/migrate-full` 대상).
5. credential, 실서버 주소, 실데이터를 문서에 넣지 않는다.

[1단계 — 세팅]
- working tree가 clean인지 확인하고, clean일 때만 브랜치 생성: `feature/ai-migration-<기능명>`
- `docs/migration/<기능명>/`에 Light 템플릿 3개를 복사한다 (프로젝트 내 `docs/migration/_templates-light/` 또는 kit의 `templates/migration-docs-light/`). 못 찾으면 01_Analysis.md / 02_Spec.md / 03_Result.md를 같은 역할로 직접 생성한다. 02_Spec.md에는 반드시 "사람이 결정할 것" 체크리스트, "범위 제외", "구현 승인" 체크박스(`- [ ] 위 계약대로 구현을 승인한다`) 섹션을 포함한다.
- 사용한 Kit 버전(commit hash 또는 zip 이름)을 01_Analysis.md 맨 위에 기록한다.

[2단계 — 분석]
- 레거시에서 추적: 진입점(URL/컨트롤러) → 호출 흐름 → 쿼리 조건(필터/삭제/상태) → 정렬 → 페이징 → 숨은 규칙(기본값, 예외 처리, 조용한 실패).
- Target 프로젝트에 같은 기능이 이미 있는지 확인한다.
- 결과를 `01_Analysis.md`에 기록한다. "미확인" 목록을 반드시 포함한다.

[3단계 — 스펙 초안]
- `02_Spec.md`에 이관 후 동작 계약 초안: API, 요청/응답, 빈 결과/에러, 정렬/페이징 기본값.
- 차이·모호한 지점은 "사람이 결정할 것" 체크리스트로. 임의로 결정하지 않는다.
- `docs/conventions/`에서 Human Decision이 Approved인 컨벤션을 찾아 반영한다. Draft는 binding으로 쓰지 않는다. 없으면 "사람이 결정할 것"에 "컨벤션 미등록 — /migrate-conventions 권장"을 추가하고 반복 패턴 기반 초안만 제시한다.
- "범위 제외"를 명시한다. 구현 승인 체크박스는 체크하지 않은 상태로 둔다.

[종료 — 여기서 이 커맨드는 끝난다]
1. 분석 요약 3~5줄
2. "사람이 결정할 것" 목록
3. 안내: "docs/migration/<기능명>/02_Spec.md를 검토하세요. 결정 항목을 채우고 구현 승인 체크박스를 체크한 뒤 /migrate-implement <기능명>을 실행하면 구현이 시작됩니다."
를 출력하고 사용자의 응답을 기다린다. 어떤 경우에도 이 턴에서 구현으로 넘어가지 않는다.
