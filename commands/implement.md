승인된 이관 스펙을 구현한다. 기능명: $1 (비어 있으면 현재 브랜치 `feature/ai-migration-<기능명>`에서 추출하고, 그래도 없으면 사용자에게 물어본다.)

[승인 확인 — 통과 못 하면 즉시 종료]
1. `docs/migration/<기능명>/02_Spec.md`를 읽는다.
2. "구현 승인" 섹션의 체크박스가 실제로 체크(`- [x]`)되어 있는지 확인한다.
3. 체크되어 있지 않으면 구현하지 않는다. "02_Spec.md의 구현 승인 체크박스가 아직 체크되지 않았습니다. 검토 후 체크하고 다시 실행해주세요."를 출력하고 종료한다.
4. "사람이 결정할 것" 항목 중 답이 비어 있는 것이 있으면 목록을 보여주고 종료한다.

[구현 직전 — binding 컨벤션 재주입]
- `docs/conventions/binding-rules.md`(또는 binding-rules-template 기반 문서)가 **Approved**면 지금 다시 읽는다. 이 규칙들이 이번 구현의 리젝 기준이다.
- 없거나 Draft면 binding 없이 진행하되, 결과에 "binding 컨벤션 미등록"을 기록한다.
- 그 외 Approved 컨벤션(구조/네이밍/응답 형식/예외/날짜/페이징)도 그대로 따른다. Draft는 binding으로 쓰지 않는다. 컨벤션과 스펙이 충돌하면 멈추고 질문한다.

[구현 — 2-pass로 진행]
- **Pass 1 (동작)**: 승인된 계약 범위만 구현하고 테스트를 통과시킨다. "범위 제외"는 건드리지 않는다.
- **Pass 2 (정리)**: Pass 1이 끝난 뒤 반드시 별도 pass로 정리한다 — ① 책임 분리(한 클래스/메서드가 조회·집계·외부호출·변환을 겸하지 않는가) ② 이름(의미 없는 이름, magic value) ③ null 흐름(실패를 null로 삼키지 않는가, 원인 로그 보존) ④ 중복/추상화(복붙 중복, 또는 성급한 추상화).
- 스펙에 없는 동작이 필요해지면 구현하지 말고 멈춰서 질문한다.
- 레거시 경로는 여전히 read-only다.

[검증]
- 테스트를 함께 작성하고 실행한다. 실패는 그대로 보고한다 — 숨기거나 assertion을 약화하지 않는다.
- `03_Result.md`에 기록: 변경 파일 / 실행한 테스트와 결과 / 검증 못 한 것 / 롤백 방법(브랜치 삭제) / **binding 규칙별 `지켰음`·`예외(사유)` 대조표** / **Pass 2 정리 내역(항목별 한 줄)**.

[종료]
- commit까지 하고 멈춘다. **push와 MR 생성은 사용자 지시가 있을 때만** 한다.
- 변경 요약과 검증 결과, 다음 권장 액션 1개를 출력한다.
