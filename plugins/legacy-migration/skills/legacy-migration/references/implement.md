# 구현 절차

## 0. 승인 확인 — 통과 못 하면 [멈춤 보고]로 종료

- Light: `docs/migration/<기능명>/02_Spec.md`의 구현 승인 체크박스가 `- [x]`인지 확인한다.
- Full: `03_Plan.md`의 `Implementation Permission: Granted*` 라인과 이번에 구현할 `IMPL-API-NNN` task를 확인한다.
- "사람이 결정할 것" 미응답 항목이나 `Open` 상태 Open Question이 있으면 목록을 보여주고 종료한다.

## 1. 착수 블록 — 코드를 만지기 전에 이 블록을 채워 출력한다

```
[구현 착수]
- 이번 묶음: IMPL-API-001, IMPL-API-002 (Full) / 승인된 스펙 전체 (Light)
- 건드릴 파일·패키지: <경로 목록>
- binding 컨벤션: <Approved binding-rules 핵심 요약, 미등록이면 "미등록">
- 멈춤 조건 해당 없음 확인: 기존 코드 교체·삭제 없음 / 스펙 밖 없음 / 공유 코드 없음
- 이 턴의 끝: [완료 보고]
```

**착수 블록이 이 턴의 계약이다.** 블록에 없는 파일을 만져야 하는 상황이 오면 그게 멈춤 조건이다.

## 2. 구현 — 세로 슬라이스

- task 하나(= API 하나)를 domain → repository → controller → 테스트까지 **한 번에 완성**한다.
- 착수 블록의 모든 task를 **중간 보고 없이 연속으로** 처리한다.
- 완성 후 정리 pass를 한 번 돈다: 책임 분리 · 이름(비즈니스 의도 — `data`/`result`/`temp`/`util`류 대신 도메인 언어) · 조건문(보호 절·빠른 반환) · null 흐름(실패의 원인 로그 보존, 명시적 실패 타입) · 추상화(필요가 확인된 것만).
- 주석은 이유·제약을 설명할 때만 쓴다. 리팩터링은 착수 블록 범위까지만 한다.

## 3. 테스트와 기록

- 성공 경로 + 빈 결과 + 잘못된 입력 + 외부 실패를 검증한다.
- Light: `03_Result.md`에 기록. Full: `04_Implement.md`(2-Pass·binding 대조)와 `05_Validate.md`(AC 단위)에 기록하고 tasks.md를 체크한다.
- commit까지 하고 멈춘다.

## 4. 턴 종료 — 두 형식 중 하나로만 끝난다

```
[완료 보고]
- 변경 사항: <파일별 요약>
- 설계 판단: <책임 배치와 구조를 이렇게 정한 이유>
- 검증 결과: <실행한 테스트와 결과>
- binding 대조: <규칙별 지켰음/예외(사유)>
- 남은 위험: <확인 못 한 것>
```

```
[멈춤 보고]
- 멈춤 조건: <6가지 중 해당 항목>
- 발견: <파일:라인 인용>
- 질문: <사람이 결정할 것>
```

## 예시

❌ **이렇게 하지 않는다** (계층 쪼개기, 선언으로 종료, 형식 없음):

> core domain 4개 클래스를 추가했습니다. 다음으로 repository를 전환하고 목록 API부터 연결하겠습니다.

✅ **이렇게 한다**:

> [구현 착수]
> - 이번 묶음: IMPL-API-001~003 …
>
> *(3개 API를 domain→repo→controller→테스트까지 구현, 중간 보고 없음)*
>
> [완료 보고]
> - 변경 사항: VisitReviewController 외 9개 파일 …
> - 검증 결과: 테스트 14개 통과(빈 결과·잘못된 입력 포함) …
