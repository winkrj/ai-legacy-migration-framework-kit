AI Legacy Migration Framework Kit v0.2

Legacy Migration Kit 사용 가이드

레거시 기능을 AI가 분석해 스펙 문서로 만들고, 당신이 승인한 범위만 새 프로젝트에 구현하는 워크플로우. 승인 전 구현은 도구 수준에서 차단됩니다.

0. 핵심 개념 — 3가지 불변 규칙

  • 레거시는 read-only — AI는 레거시 repo의 파일을 절대 수정하지 않습니다.
  • 계약은 사람 승인 후 구현 — 스펙 문서(02_Spec.md)의 승인 체크박스는 사람만 체크합니다. 체크 전에는 훅이 코드 수정을 차단합니다.
  • 격리 브랜치 — 모든 이관은 feature/ai-migration-<기능명> 브랜치에서만 진행됩니다. 롤백 = 브랜치 삭제.

1. 설치 (최초 1회)

Claude Code

/plugin marketplace add winkrj/ai-legacy-migration-framework-kit
/plugin install legacy-migration@legacy-migration-kit

커맨드 5종 + 스킬 + 승인 게이트 훅이 설치됩니다.

Codex — 플러그인

codex plugin marketplace add winkrj/ai-legacy-migration-framework-kit

이후 Codex 앱/CLI의 Plugins 화면에서 Legacy Migration Kit을 찾아 설치합니다. 설치하면 자연어 요청("이 기능 이관해줘")에 스킬이 자동으로 같은 워크플로우를 적용합니다.

Codex — 커스텀 프롬프트 (플러그인 대신)

npx --yes github:winkrj/ai-legacy-migration-framework-kit

~/.codex/prompts/에 커스텀 프롬프트 5종이 설치됩니다. 타겟 프로젝트 루트에 kit의 agent/codex/AGENTS.md를 복사(또는 병합)하는 것을 권장합니다.

Codex에서는 승인 게이트 훅의 동작이 아직 검증되지 않았습니다. 분석 세션과 구현 세션을 분리해서 사용하세요.

2. 명령어 치트시트

Claude CodeCodex하는 일
/legacy-migration:conventions [참고경로]/migrate-conventions [참고경로]컨벤션 등록 (아래 3가지 방식)
/legacy-migration:start 기능명 레거시경로/migrate-start 기능명 레거시경로레거시 분석 + 스펙 초안 → 멈춤
/legacy-migration:implement 기능명/migrate-implement 기능명승인 확인 후 구현 + 테스트 + commit
/legacy-migration:validate 케이스명/migrate-validate 케이스명이관 문서 구조 검사 (Validator CLI)
/legacy-migration:full 기능명 레거시경로/migrate-full 기능명 레거시경로결제·인증·PII·공유 코드용 Full 모드

커맨드 없이 "이 기능 이관해줘"라고만 해도 (Claude Code에서는) 스킬이 같은 워크플로우를 자동 적용합니다.

3. 프로젝트 준비 — 컨벤션 등록 (프로젝트당 1회, 권장)

이관 결과물이 팀 스타일을 따르게 하려면 먼저 컨벤션을 등록하세요. 세 가지 방식을 지원합니다.

방식사용법
팀 가이드/참고 프로젝트에서 추출/legacy-migration:conventions ~/work/team-guide-project — 반복 패턴과 반례를 근거로 초안 생성
직접 입력커맨드 실행 후 대화로 규칙 제시 → 템플릿 구조로 정규화
현재 프로젝트에서 추출인자 없이 실행 → AI가 방식을 물어봄

결과는 docs/conventions/에 Draft로 저장됩니다. 검토 후 문서의 Human Decision을 Approved로 바꿔야 이후 스펙/구현이 binding rule로 사용합니다. 패턴이 갈리는 지점은 AI가 결정하지 않고 Open Question으로 물어봅니다.

4. 이관 플로우 (기능당 반복)

  1. YOU이관 시작

    /legacy-migration:start 공지사항목록 ~/work/legacy-admin

    AI브랜치 생성 → 레거시 추적(진입점→호출 흐름→쿼리 조건→정렬/페이징→숨은 규칙) → docs/migration/공지사항목록/에 01_Analysis.md(미확인 목록 포함)와 02_Spec.md(동작 계약 초안 + "사람이 결정할 것" + 범위 제외) 작성 후 멈춥니다.

  2. YOU스펙 검토와 승인

    02_Spec.md를 열어 ① "사람이 결정할 것" 항목에 답을 채우고 ② 맨 아래 구현 승인 체크박스를 체크합니다.

    - [x] 위 계약대로 구현을 승인한다 (승인자: 홍길동 / 날짜: 2026-07-06)
    승인 게이트: 이 체크박스가 체크되기 전에는 훅이 docs/migration/ 밖의 모든 파일 수정을 차단합니다. AI가 규칙을 "깜빡"해도 물리적으로 구현할 수 없습니다.

  3. YOU구현 지시

    /legacy-migration:implement 공지사항목록

    AI승인 체크박스와 결정 항목을 확인 → 승인된 계약 범위만 구현(Approved 컨벤션 준수) → 테스트 작성·실행(실패는 그대로 보고) → 03_Result.md 기록 → commit 후 멈춥니다. 스펙에 없는 동작이 필요해지면 구현하지 않고 질문합니다.

  4. YOU결과 확인 → push/MR

    03_Result.md(변경 파일/테스트 결과/검증 못 한 것/롤백 방법)를 확인하고, 괜찮으면 "push하고 MR 만들어줘"라고 지시합니다. push와 MR은 항상 사람 지시로만 진행됩니다.

5. Full 모드 — 고위험 케이스

다음 중 하나라도 해당하면 Light 모드로 진행하지 않습니다: 결제 · 인증 · 개인정보(PII) · 여러 기능이 공유하는 코드 · production cutover 판단. AI도 분석 중 이런 코드를 만나면 스스로 멈추고 보고합니다.

/legacy-migration:full은 8문서(Discover→Specify→Plan→Implement→Validate→Archive) + OpenSpec + 단계별 Human Gate로 진행하며, 각 단계 산출물마다 /legacy-migration:validate로 문서 검사를 돌립니다.

6. 문서 검사 (Validator)

/legacy-migration:validate 공지사항목록

내부적으로 npx github:winkrj/legacy-migration-validator-cli를 실행해 필수 문서/섹션/필드, 상태 어휘, 민감정보 패턴, 구현 경계를 검사하고 reports/에 리포트를 남깁니다. 별도 설치나 권한이 필요 없습니다. Validator 통과 = 문서 구조 OK일 뿐, 도메인 정확성 보장이 아닙니다.

7. 자주 묻는 질문

AI가 승인 전에 코드를 만들려고 하면?

Claude Code에서는 훅이 자동 차단하고 AI에게 멈추라는 메시지가 전달됩니다. Codex에서는 "멈춰. 스펙 승인 전이야"라고 답하면 됩니다 — 프로토콜 위반입니다.

"사람이 결정할 것" 항목이 무슨 뜻인지 모르겠어요

maintainer에게 물어보세요. 애매한 걸 사람에게 묻는 것이 이 kit의 정상 동작입니다. 임의로 아무 답이나 채우고 승인하는 것이 가장 위험합니다.

이관을 되돌리고 싶어요

모든 작업은 feature/ai-migration-<기능명> 브랜치에 격리되어 있습니다. 브랜치를 삭제하면 됩니다. 구체적 롤백 방법은 각 케이스의 03_Result.md에 기록됩니다.

컨벤션 없이 시작해도 되나요?

됩니다. 이 경우 AI가 타겟 프로젝트의 반복 패턴을 참고해 초안 수준으로만 반영하고, 스펙의 "사람이 결정할 것"에 컨벤션 미등록을 표시합니다. 팀 표준을 지키려면 등록을 권장합니다.

테스트가 실패한 채로 끝났어요

정상 동작입니다 — kit은 실패를 숨기거나 assertion을 약화하는 것을 금지합니다. 03_Result.md의 실패 내역을 보고 스펙 문제인지 구현 문제인지 판단한 뒤 재지시하세요.
